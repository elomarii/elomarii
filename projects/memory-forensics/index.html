Memory Forensics

Doing some kung-fu to explore the inside of a memory dump of a compromised system.

Posted Sep 14, 2024
By Ibrahim
6 min read

The following is inspired by TryHackMe’s Forensics room.

I. Intro

In this project, we’ll take a deep dive into memory forensics using Volatility3.

Memory forensics is forensic analysis of a computer’s memory dump. Its primary application is investigation of advanced computer attacks which are stealthy enough to avoid leaving data on the computer’s hard drive. Consequently, the memory (e.g. RAM) must be analyzed for forensic information1.

Volatility is an open-source memory forensics framework for incident response and malware analysis. It is written in Python and supports Microsoft Windows, Mac OS X, and Linux2. In this project, we’ll be using Volatility 3 to alanyze a memory dump of a compromised system, and learn the tool along the way.

II. Volatility

Let’s start with a simplified overview of how Volatility works. We need to understand four fundamental concepts: Layers, Templates, Objects, and Symbol tables.

Layouts

A layout in Volatility defines the structure and organization of memory for a specific operating system or platform. Different operating systems (e.g., Windows, Linux, macOS) have different memory structures and Volatility needs to understand these to extract meaningful information. Thus, layouts provide a framework for parsing memory based on the OS’s kernel version, architecture (x86, x64), and any other factors that influence memory organization. For example: A Windows 10 x64 memory dump will have a different layout than a Linux x86 memory dump.

Layers

A layer is a logical or physical segment of memory data that Volatility reads, interprets, and processes. Layers often need to be combined or stacked to handle different types of memory access or address translation. Its main purpose is to allow for a modular approach to handling different memory representations. The common examples of layers include

  • Physical layer: raw, unprocessed physical memory data from the dump.
  • Paged memory layer: Deals with operating system-specific paging, usch as handling page tables and translating virtual addresses into physical ones.
  • ELF layer / PE layer: Interprets binary executable formats like ELF (Linux) or PE (Windows) to understand how an executable maps into memory.

Templates and Objects

A template describes the format of specific data structures within memory. These structures are defined by the operating system or the software being analyzed (e.g., a process control block, thread structures, file handles). By providing the blueprint to interpret specific objects or data structures, they define the fields, types, and relationships within an object in memory. Example: A template for the EPROCESS structure in Windows defines how a process is laid out in memory, with fields like ProcessId, ParentProcessId, ThreadList, and others. Volatility can then parse a memory dump using this template to extract process-related information.

On the other hand, an object is an instance of a particular structure or data type found within the memory dump. Volatility 3 uses templates to convert raw memory data into objects that it can interpret and analyze.

Symbol tables

A symbol table is a data structure used by a compiler or interpreter to store information about the names, or symbols, used in a program. It is essential for managing the mapping between identifiers (like variable names, function names, and objects) and the information associated with them (such as their data types, memory locations, or scopes). These tables are produced as debugging information alongside the compilation of programs.
